# 小组分工与集成方案

## 1. 怎么拆分

我们把代码按依赖关系拆成三块，每人都有一块能独立验收的主要代码。文档、截图和汇报会一起做，但不拿它们代替编码工作量。最后由组长把三部分接起来：

```text
成员一：完整俄罗斯方块玩法（Game Model）
              ↓ 调用组长提供的 ScoreSystem
成员二：界面、输入与本地场景（View + Input）
              ↓ 提供绘制函数和按钮
组长：计分系统、应用总控与局域网（Scoring + Controller + Network）
```

这样拆的关键是依赖不要倒过来：规则核心不导入 UI 或网络；UI 只读取游戏状态，不自己改分数；网络只传快照，不碰 Pygame 画布。

---

## 2. 成员一：除计分外的全部俄罗斯方块功能

### 负责内容

- 编写 `tetris/config.py` 中的棋盘参数、方块颜色和比赛参数。
- 编写 `tetris/core.py` 中除 `ScoreSystem` 外的所有俄罗斯方块代码。
- 编写方块、棋盘、移动、旋转、下落、锁定、消行、暂存、结束判定和快照测试。

文档内容不计入该成员工作量。

### 必须实现的功能

1. 10×20 棋盘和七种 Tetromino 模板。
2. `Piece` 坐标和旋转状态。
3. 7-bag 随机序列和可选随机种子。
4. 边界、堆叠碰撞检测。
5. 左右移动、软降、硬降和自动下落。
6. 顺/逆时针旋转和简化墙踢。
7. 暂存、五格预览队列和幽灵落点。
8. 生成下一个活动块并维护预览队列。
9. 0.5 秒落地锁定。
10. 完整行检测、删除和顶部空行补充。
11. 游戏结束与暂停状态。
12. 联机显示快照。

成员一在软降、硬降和消行发生时，只负责识别事件并调用组长提供的计分接口，不自行计算任何分值。

### 具体代码清单

成员一必须亲自实现以下代码，不得只提交说明文档：

```text
tetris/config.py
  全部常量和 COLORS 映射

tetris/core.py
  SHAPES
  rotate_cells()
  Piece
  Bag
  TetrisGame（除读取 ScoreSystem 状态外的全部实现）
  TetrisGame.__init__()
  TetrisGame.fall_interval
  TetrisGame.valid()
  TetrisGame.move()
  TetrisGame.rotate()
  TetrisGame.soft_drop()
  TetrisGame.ghost_y()
  TetrisGame.hard_drop()
  TetrisGame.swap_hold()
  TetrisGame.spawn()
  TetrisGame.lock()
  TetrisGame.update()
  TetrisGame.snapshot()

tests/test_core.py
  除 ScoreSystem 单元测试之外的全部俄罗斯方块测试
```

按当前实现估算，这部分大约有 240～300 行有效 Python 代码（不算空行和纯注释）。

### 交付接口

- `Piece`
- `Bag`
- `TetrisGame` 全部公开接口
- `rotate_cells()`

接口签名和字段以 `12_完整接口文档.md` 为准，内部怎么实现由负责人决定。

### 验收标准

- 不初始化 Pygame 也能导入和运行。
- `python -m unittest discover -s tests -v` 全部通过。
- 相同 seed 能产生相同方块序列。
- 棋盘始终为 20×10。
- 下落、锁定、消行、快照及操作状态符合接口约定。

### 难度

中等偏高。重点是完整玩法流程、坐标边界、锁定和消行正确性，但不负责编写分数公式。

## 3. 成员二：UI、输入与本地游戏场景

### 负责内容

- 编写 `tetris/ui.py` 的全部绘制组件。
- 编写 `tetris/app.py` 中所有输入处理和画面绘制方法。
- 编写主菜单、单机界面、联机大厅、双棋盘和暂停/结果遮罩。
- 编写 Windows/macOS 中文字体加载代码。

截图、布局说明和手工验收不计入该成员工作量。

### 必须实现的功能

1. 纯色背景和统一配色。
2. 中文文字绘制及字体缓存。
3. 可悬停、可点击按钮。
4. 10×20 棋盘、网格、锁定方块、活动块和幽灵块。
5. 暂存和下一个方块预览。
6. 分数、消行、等级、连击显示。
7. 联机双方棋盘、双方分数和倒计时布局。
8. 暂停、游戏结束和比赛结果遮罩。
9. 输入映射：方向键、Z/X、空格、C、P、Esc。

### 具体代码清单

成员二必须亲自实现以下代码：

```text
tetris/ui.py
  font()
  text()
  panel()
  Button.__init__()
  Button.draw()
  Button.clicked()
  draw_block()
  draw_board()
  draw_preview()

tetris/app.py（只负责以下方法）
  background()
  menu()
  input_game()
  draw_game()
  current_cells()
  ghost_cells()
  draw_single()
  draw_battle()
  overlay()
```

这部分不只是画画面。`menu()` 要处理按钮事件，`input_game()` 要把键盘输入转成规则调用，`draw_battle()` 还要能画网络传来的对手快照。按当前实现估算，共约 220～280 行有效 Python 代码。

### 交付接口

- `font()`、`text()`、`panel()`、`draw_block()`
- `Button`
- `draw_board()`、`draw_preview()`
- `app.py` 中上述九个输入和绘制方法

成员二不直接修改 `TetrisGame.board`、`score` 等状态。用户操作必须通过 `move()`、`rotate()`、`soft_drop()`、`hard_drop()`、`swap_hold()` 等公开方法完成。

### 验收标准

- Windows 和 macOS 中文显示正常。
- 同一个 `draw_board()` 可绘制本地大棋盘和对手小棋盘。
- 空暂存区不会报错。
- 活动块位于棋盘顶部之外时不会画到非法位置。
- UI 不依赖网络连接即可单独渲染。

### 难度

中等。重点是 Pygame 坐标换算、事件命中和清晰布局。

## 4. 组长：计分系统、应用总控、局域网与最终集成

### 负责内容

- 编写 `main.py`。
- 编写 `tetris/network.py` 全部网络代码。
- 独立编写 `tetris/core.py` 中的 `ScoreSystem` 计分系统。
- 编写计分系统的全部自动测试。
- 编写 `tetris/app.py` 中的程序生命周期、场景状态和联机控制方法。
- 编写 UDP/TCP 异常处理以及最终模块集成代码。

README、汇报材料、合并操作和双机手工验收不计入组长的代码工作量。

### 必须实现的功能

1. 保存总分、累计消行、等级和连击状态。
2. 软降按每格 1 分计分。
3. 硬降按每格 2 分计分。
4. 单次消除 1/2/3/4 行获得 100/300/500/800 基础分并乘等级。
5. 连续消行增加 `combo × 50`，无消行时中断连击。
6. 每累计消除 10 行提升一级。
7. 初始化 Pygame、窗口、时钟和 60 FPS 主循环。
8. 在 `menu/single/network/battle` 四个场景间切换。
9. 调用成员一玩法接口和成员二绘制接口。
10. UDP 45678 房间发现和房间响应。
11. TCP 45679 建房、连接、昵称交换和状态同步。
12. 网络线程与 Pygame 主线程通过队列隔离。
13. 主机发起比赛，双方按统一时间开始。
14. 每约 80ms 发送本地快照。
15. 双方实时显示对手棋盘与分数。
16. 60 秒到时判定胜、负、平局。
17. 对手断开、搜索失败、连接失败和退出清理。

### 具体代码清单

组长还要把自己的 `ScoreSystem`、成员一的玩法逻辑和成员二的 UI/输入方法接入主循环。按当前实现估算，这部分约 350～430 行有效 Python 代码，难点主要在联调和异常处理。

### 交付接口

- `LanSession` 和 `local_ip()`
- `TetrisApp`
- `ScoreSystem` 全部接口
- JSON Lines 网络协议
- 可直接运行的 `main.py`

### 验收标准

- 单机模式在断网状态下完整可用。
- 搜索和连接不会冻结 Pygame 窗口。
- 本机两个进程能够连接、开始比赛并交换快照。
- 两台局域网设备能够完成 60 秒比赛。
- 退出房间后端口可再次使用。
- 集成时只调用另外两名成员约定的公开接口，不依赖其私有实现。

### 难度

较高，作为组长任务。它同时涉及计分状态设计、线程、TCP 字节流拆包、UDP 广播、场景状态、时间同步、错误处理和最终模块集成。

## 5. 推荐开发顺序

1. 全员先确认 `12_完整接口文档.md`，接口冻结后再编码。
2. 组长先完成独立 `ScoreSystem`；成员一通过其接口实现完整玩法；成员二使用固定二维数组开发 UI。
3. 成员一完成 `TetrisGame` 后，组长接入应用主循环和网络传输。
4. 成员一、二分别提交后，组长接入 `TetrisApp`。
5. 组长合并成员二负责的 `app.py` 方法，先完成本机双进程测试，再进行两台设备联机测试。
6. 发现接口问题时先修改文档、三人确认，再修改代码，避免各自实现不同版本。

## 6. Git 协作建议

- 成员一分支：`feature/core`
- 成员二分支：`feature/ui`
- 组长分支：`feature/network-app`
- 集成分支：`develop`
- 稳定提交进入：`main`

每次合并请求至少包含：改动说明、接口是否变化、测试命令、测试结果。不要提交 `__pycache__/`、虚拟环境或 IDE 缓存。

## 7. 工作量概览

| 部分 | 主要文件 | 参考代码量 | 难度 | 主要风险 |

## 8. `core.py` 接口所有权

成员一和组长都会编写 `core.py`，所以先把各自负责的接口分清：

- 组长只实现 `ScoreSystem` 及其三个公开计分方法。
- 成员一实现 `Piece`、`Bag`、`TetrisGame` 及全部玩法方法。
- `TetrisGame.soft_drop()` 只在移动成功后调用 `scoring.award_soft_drop()`。
- `TetrisGame.hard_drop()` 计算距离后调用 `scoring.award_hard_drop(distance)`。
- `TetrisGame.lock()` 负责找出消行数，然后调用 `scoring.apply_line_clear(count)`。
- 成员一不得在 `TetrisGame` 中重复写分值公式；组长不得在 `ScoreSystem` 中操作棋盘或方块。
- 双方测试放在同一测试文件时，应按测试类或方法名前缀区分所有权。

双方以 `ScoreSystem` 的公开接口为唯一连接点，禁止读取对方私有实现。

## 9. `app.py` 共同修改规则

成员二和组长都会改 `app.py`，但负责的方法不重叠。为了少解冲突，约定按下面的顺序做：

1. 组长先创建只包含空方法和正确签名的 `TetrisApp` 骨架。
2. 成员二只填写分配给自己的九个方法，不修改构造字段和网络方法。
3. 组长只填写其余方法，不改成员二已经确认的绘制接口。
4. 合并时按“方法”为单位解决冲突，不允许直接用一个人的完整文件覆盖另一个人的代码。

如果想完全避免两人同时改一个文件，可以让成员二新增 `tetris/screens.py`，把九个方法拆成接收 `app` 或状态参数的函数，`app.py` 只负责调用。文件可以这样调整，但 `12_完整接口文档.md` 里的数据格式不能跟着变。
